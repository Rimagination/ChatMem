# ChatMem Compact Layout And Folding Design

**Date:** 2026-04-08

## Goal

在保留当前 ChatMem 双栏工作台结构的前提下，进一步提升阅读效率：

- 左侧对话列表改成更紧凑的列表式导航
- 右侧长助手消息默认折叠，避免单条输出撑满整页
- 工具调用和长结构化输出允许独立折叠，降低阅读噪音

本次设计不改动数据来源、不改动迁移功能、不改动顶部“对话文件位置 / 复制位置 / 复制恢复命令”的已上线方案。

## User Need

当前界面已经比旧版本更清晰，但仍有两个阅读问题：

1. 左侧会话列表仍然偏“卡片化”，单条占用高度过大，不利于快速扫描多个会话
2. 助手输出一旦很长，会把消息流垂直拉得很长，影响上下文定位和查找

用户希望它更接近主流 AI 聊天工具：

- 左栏像目录列表，只保留最关键的标题信息
- 长回复默认收起，只在需要时展开全文

## Product Decision

### Primary change set

本次只做两类核心调整：

1. **紧凑侧栏**
2. **默认折叠的助手消息**

这是一次阅读体验优化，而不是信息架构重做。

### What stays unchanged

- 仍然保留 agent 切换 tabs
- 仍然保留搜索框
- 仍然保留右侧统计卡、文件变更区、迁移/删除按钮
- 用户消息仍然保持完整显示，不默认折叠
- 顶部“对话文件位置”信息条继续保留

## Visual Direction

### Visual thesis

从“轻卡片工作台”继续收紧到“高信息密度聊天阅读器”。

整体方向：

- 左侧更轻、更扁平、更像会话目录
- 右侧把视觉焦点继续收束到消息流本身
- 弱化次要元数据，强化标题、时间、折叠状态这三类真正影响阅读速度的信息

### Density thesis

空间优先分配给“更多对话”和“更短消息预览”，而不是分配给重复元数据标签、装饰性 pills 或大面积留白。

## Left Sidebar Design

### Row structure

左侧每个会话条目改成紧凑列表行，而不是高卡片。

每条只保留：

- 会话标题第一行
- 必要时显示一行较弱的项目目录或来源提示
- 右侧显示相对时间

不再把消息数、文件数做成显眼的圆角 pills。若保留，只允许退到更弱的一行文本，不能再主导视觉层级。

### Title behavior

- 标题单行显示
- 超出宽度时使用省略号
- 默认只展示摘要前半段，不追求完整可见

这样能让左栏在同一屏里看到更多会话。

### Selected state

选中项继续有明显状态，但改为更克制的高亮条目效果：

- 更轻的背景
- 更细的边框或左侧强调线
- 减少整块大面积着色

目标是“明确选中”而不是“像一张被展开的大卡片”。

## Conversation Stream Design

### Folding model

所有 `assistant` 消息默认进入“预览折叠态”。

默认折叠态行为：

- 显示约 5 行正文预览
- 底部显示 `展开全文`
- 点击后切换到完整内容，并显示 `收起`

`user` 消息默认保持完整，不折叠。因为用户自己的输入通常较短，且在追溯上下文时需要直接看到完整提问。

### Preview behavior

折叠预览必须满足：

- 能快速判断这一条回复在讲什么
- 但不会让完整长文一次性占据整屏

预览方式使用 CSS 行数裁剪而非截断字符串，这样：

- 不破坏原始文本内容
- 展开时无需重新计算文本
- 能保持代码块、段落和中文文本的自然断行

### Toggle affordance

折叠切换控件使用明确文案：

- `展开全文`
- `收起`

控件位置放在消息内容底部，样式弱于主按钮，但必须足够明显，不能隐藏成纯图标。

## Tool Call Folding

### Secondary folding layer

工具调用和长结构化输出继续显示，但作为消息下的次级内容默认收起。

目标：

- 主消息先让用户看懂“这条助手回复的大意”
- 工具调用、JSON 参数、长输出作为按需展开的信息

### Behavior

每个 tool call 块允许独立展开/收起：

- 默认展示工具名与状态
- 展开后再展示 input/output

这可以显著降低长 JSON 把页面撑长的问题。

## Information Hierarchy

### Message priority

右侧阅读区的优先级调整为：

1. 会话标题
2. 对话文件位置条
3. 用户与助手消息正文
4. 折叠控件
5. 时间、角色、工具调用细节
6. 统计信息与文件变更

这意味着时间、角色标签和次要状态都要更轻，不能再和正文抢视觉注意力。

## Frontend Scope

### Expected files

- `src/components/ConversationList.tsx`
- `src/components/ConversationDetail.tsx`
- `src/styles.css`
- `src/__tests__/ConversationList.test.tsx`
- `src/__tests__/ConversationDetail.test.tsx`

若实现需要非常少量的状态传递，也允许调整 `src/App.tsx`，但本次优先避免扩大改动面。

### State model

折叠状态优先在 `ConversationDetail` 内部维护：

- 以 message id 为 key 记录展开状态
- 切换会话后重新按默认折叠态渲染

不需要把折叠状态持久化到本地存储。

## Out Of Scope

- 不新增数据库字段
- 不新增后端接口
- 不调整迁移逻辑
- 不做全局“紧凑模式”设置项
- 不做代码块语法高亮重构
- 不做多级筛选或文件夹式侧栏分组

## Error Handling

- 如果某条消息没有正文，只显示 tool call 区块
- 如果 tool call 没有 output，展开后只显示 input
- 如果消息很短，不应显示无意义的 `展开全文`

## Testing Strategy

实现必须遵循 TDD。

至少覆盖以下行为：

- 左侧列表标题为单行紧凑结构
- 左侧列表不再渲染旧的 metadata pills
- 助手长消息默认处于折叠态
- 点击 `展开全文` 后显示完整正文，再次点击可 `收起`
- 用户消息默认不折叠
- tool call 默认折叠，仅显示名称和状态，展开后再显示详细内容

## Acceptance Criteria

- 左侧同一屏可见的会话数量明显增加
- 左侧会话条目视觉上更像列表，而不是大卡片
- 助手长消息默认不会把整页一下子撑开
- 用户能快速展开全文查看完整回复
- 工具调用默认不再抢占主阅读区高度
- 现有顶部“对话文件位置 / 复制位置 / 复制恢复命令”功能保持可用
